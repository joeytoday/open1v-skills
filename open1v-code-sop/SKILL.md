# Joey Code SOP

与 Joey 协作开发时的标准操作流程和约束规范。

## 核心原则

### 先想清楚再动手

- **显式表达假设**：动手前先说清楚你的假设。不确定的地方直接问，不要默默猜测
- **多种理解时摊开讲**：如果需求存在多种解读，列出来让用户选，不要自己替用户决定
- **该推回就推回**：如果存在更简单的方案，主动提出。需求不合理时说明理由
- **不确定就停下来**：说清楚哪里不确定，问清楚再动手

### 最小改动原则

- **尽可能少修改代码**：每次修改只解决一个具体问题，不做额外的重构或优化
- **不改动现有组件功能**：修复 bug 时不能影响已有功能的行为
- **已实现的功能和函数尽量不改动**：已验证可用的功能保持原样，不要用新方案替代
- **只删除自己引入的废弃代码**：你的改动导致的 import/变量/函数不再使用，必须清理；已有的死代码提一句即可，不要擅自删除
- **精准定位**：先充分分析根因，确认最小改动点，再动手修改
- **匹配现有风格**：即使你觉得可以写得更好，也按项目现有风格来。每一行改动都要能追溯到用户的需求

### 简单优先原则

- **不加没被要求的功能**：不做投机性开发
- **不为单次使用的代码加抽象层**：不搞"灵活性"和"可配置性"
- **不为不可能的场景加错误处理**
- **能 50 行解决的问题不写 200 行**：写完问自己"一个高级工程师会觉得这过度设计了吗？"，如果会，简化

### 逐步验证原则

- **一次只解决一个问题**：即使 issue 中包含多个问题，也要逐个解决
- **同类问题可合并**：如果多个问题本质上是同一类问题（如都是 SVG 兼容性），可以一起修复
- **等待用户确认**：问题修复后，等用户测试通过确认后，再解决下一个问题
- **不要抢跑**：不要在用户确认前就开始下一个问题的修复

## 工作流程

1. **分析问题**：阅读 issue/需求，理解问题现象和预期行为。显式列出你的假设
2. **定位根因**：通过代码搜索和阅读，找到问题的根本原因
3. **定义成功标准**：把任务转化为可验证的目标（如"写一个复现 bug 的测试，然后让它通过"），而不是模糊的"修好它"
4. **制定方案**：确定最小改动方案，向用户说明根因和修复思路。多步任务列出步骤和每步的验证方式
5. **实施修改**：执行代码修改，确保无 lint 错误
6. **等待反馈**：告知用户修改完成，等待用户测试确认
7. **确认后继续**：用户确认通过后，再处理下一个问题
8. **不主动提交代码**：修改完成后不要自动 git commit，等用户测试通过后明确要求提交时再提交

## 功能一致性原则

- **复制和发布功能必须统一**：复制（copyToClipboard）和发布（publish）走同一个处理管线
- **新增功能时**：必须同时确认复制和发布两个入口都能覆盖到
- **删除功能时**：必须同时确认复制和发布两个入口都已清理
- **验证方式**：修改后检查两个入口的调用链，确保行为一致

## 版本发布规范

- **CHANGELOG 简单说明**：只描述用户可感知的变更，不涉及技术细节（如函数名、变量名、实现方式等）
- **版本号**：小版本迭代（patch），同步更新 `manifest.json` 和 `package.json`
- **Release tag**：不包括 `v` 开头，直接使用版本号（如 `2.3.7`）
- **Release 产物**：`main.js`、`styles.css`、`manifest.json` 三个文件 + 压缩包
- **Release Note**：与 CHANGELOG 保持一致，简单说明变更内容

## Issue 回复规范

- **简洁为主**：说清楚修了什么、版本号、PR 链接，不废话
- **不解释技术细节**：不写根因分析、不贴代码、不说 " 通过 xxx 方法实现 "
- **用户视角**：只描述用户能感知到的变化（如 " 代码块正常换行了 "），不描述实现方式
- **格式参考**：

```
已修复，发布在 [x.x.x](release链接)。
- 修复了 xxx
- 修复了 yyy
PR: #xx
```

- **不要**：感谢反馈、抱歉给您带来不便、如有问题请随时联系、希望这次更新能改善您的体验

## 禁止事项

- ❌ 一次性修复多个不相关的问题
- ❌ 在修复 bug 时顺手重构代码
- ❌ 未经用户确认就开始下一个修复
- ❌ 改动与问题无关的代码
- ❌ 添加用户没有要求的新功能
- ❌ 只在复制或发布其中一个入口添加/删除功能，而遗漏另一个


## 版本管理

本技能修改后，须遵循 `open1v-skills-manage` 规范完成版本号更新。
